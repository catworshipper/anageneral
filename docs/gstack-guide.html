gstack — Project Guide

How to use the gstack Claude Code skill suite on this project.

alpacaplayhouse.com · static HTML/JS, no server-side code · gstack installed at ~/.claude/skills/gstack

▸ The golden rule

Use /browse for all web browsing — real headless Chromium, real clicks, real screenshots, purpose-built for QA and automation.

Never use mcp__claude-in-chrome__* / mcp__Claude_in_Chrome__* tools in this project. This is enforced in CLAUDE.md's ## gstack section.

01What is gstack

gstack is a free, MIT-licensed suite of Claude Code skills built by Garry Tan (YC). It bundles a real headless-browser tool (/browse) with ~30 slash-command "specialists" that mirror a small engineering team — a product interrogator, an eng manager, a designer, a code reviewer, a QA lead, a security officer, and a release engineer — so a solo builder can run a full think → plan → build → review → test → ship → reflect sprint from inside one Claude Code session.

Everything is a Markdown skill file plus the bundled browser binary. No paid tier, no account required to use it locally.

02How to invoke a skill

Two ways:

  • Type the slash command directly in Claude Code, e.g. /qa https://alpacaplayhouse.com
  • Tell a fresh session in plain language — useful when kicking off a sub-agent or a new terminal: Load gstack. Run /qa https://alpacaplayhouse.com

Skills chain naturally — most read the output of the skill before them (e.g. /office-hours writes a design doc that /plan-ceo-review reads).

03Project playbook — mapped to our workflow

This project is a static HTML/JS site with no server-side code (browser → Cloudflare Pages → Supabase). Push to main auto-deploys — that changes how a few skills apply here versus a typical backend project. Notes are called out below.

Ship / deploy

/ship → /land-and-deploy

/ship syncs main, runs tests, audits coverage, pushes, and opens a PR. /land-and-deploy merges it, waits for CI + the Cloudflare Pages deploy, and verifies production health — one command from "approved" to "verified live."

/ship > PR opens against main /land-and-deploy

Since this repo pushes straight to main per CLAUDE.md's mandatory behaviors, /ship's PR step is optional here — many changes just get pushed directly and verified with /qa after.

Pre-landing review

/review · /codex

/review is a staff-engineer pass over the diff — finds bugs that pass CI but blow up in prod, auto-fixes the obvious ones. /codex gets an independent second opinion from OpenAI's Codex CLI on the same diff.

/review /codex

Test a live page

/qa · /qa-only + /browse

/qa opens a real browser against a URL, clicks through flows, finds bugs, fixes them, and writes a regression test. /qa-only runs the same methodology but reports without touching code. /browse is the underlying tool for one-off manual checks.

/qa https://alpacaplayhouse.com/residents/ /qa-only https://alpacaplayhouse.com/spaces/

This is the highest-value skill for this project — a static site with real Supabase-backed forms and portals is exactly what a headless browser QA pass is built to catch.

Debug

/investigate

Systematic root-cause debugging — Iron Law: no fixes without investigation. Traces data flow, tests hypotheses, stops after 3 failed attempts instead of thrashing.

/investigate the reservation form on /spaces/ silently fails to submit

Security

/cso

OWASP Top 10 + STRIDE threat model with an 8/10+ confidence gate and independent finding verification — each finding ships with a concrete exploit scenario.

/cso

Worth running against docs/CREDENTIALS.md-adjacent code and anywhere Supabase row-level security is load-bearing, given the backend here is still lightly provisioned.

Post-ship

/document-release · /retro

/document-release reads the diff and updates every doc that drifted (README, CLAUDE.md, this guide, etc.) — catches stale docs automatically. /retro runs a team-aware weekly retrospective.

/document-release /retro

Plan first

/office-hours → /autoplan

/office-hours is the forcing-question interrogation that turns a vague feature request into a real design doc before any code is written. /autoplan then runs CEO → design → eng review on that doc automatically, surfacing only the taste decisions for approval. For a single-lens pass, use /plan-ceo-review or /plan-eng-review directly.

/office-hours > describe the feature /autoplan

Design

/design-shotgun · /design-review · /design-html · /plan-design-review

/design-shotgun generates 4-6 mockup variants and opens a comparison board for you to pick and iterate on. /design-html turns an approved mockup into real, responsive HTML/CSS — matching this project's "single self-contained page" convention well. /design-review / /plan-design-review audit AI-slop and design quality before or after shipping.

/design-shotgun a warm-paper style internal report page /design-html /design-review
Less relevant here: skills built around a backend or app-store surface — /benchmark (Core Web Vitals still apply, but this is a lightweight static site), /plan-devex-review / /devex-review (matter more for SDKs/CLIs than an internal admin tool), and the /ios-* family (no native app in this project).
